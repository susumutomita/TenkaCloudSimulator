# Compatibility reports

This directory contains deterministic, reviewable release evidence. The
capability manifest is generated from provider code. The catalog coverage
report is generated by comparing that manifest with a pinned
TenkaCloudChallenge checkout.

Manifest `fidelity` values are non-empty canonical sets (`L0` through `L4`),
not ordered maximum levels. Requirements retain their full canonical sets;
coverage requires every exact member and reports the complete required and
implemented sets. Capability matching also requires the exact provider and
engine; an implementation for another engine never covers the requirement.

Regenerate both artifacts before a release:

```bash
bun tools/capability-manifest/src/bin.ts \
  --source-commit <clean-simulator-commit> \
  --output reports/capabilities.json
bun tools/catalog-scanner/src/cli.ts \
  --catalog /path/to/TenkaCloudChallenge \
  --catalog-commit "$(git -C /path/to/TenkaCloudChallenge rev-parse HEAD)" \
  --capabilities reports/capabilities.json \
  --simulator-version <exact-capability-manifest-version> \
  --output reports/catalog-coverage.json
bunx --bun biome format --write \
  reports/capabilities.json \
  reports/catalog-coverage.json
```

`catalog-coverage.json` follows
`contracts/schemas/capability-report.schema.json`. Its `catalogCommit` is the
immutable 40-character commit used for the scan. Generation rejects a different
HEAD or uncommitted changes within the catalog scope, and the requested simulator
version must exactly match the capability manifest version. The manifest version
contains the clean Simulator source commit, and every catalog source byte used by
the scan must match a tracked blob at `catalogCommit`; ignored or index-hidden
artifacts are rejected. `reportHash` is the
SHA-256 of the recursively key-sorted canonical report payload with only
`reportHash` omitted. The root `requirements` list contains the binding
requirements that decide compatibility. Scanner inventory, catalog diagnostics,
problem metadata, and non-gating IAM authorization inventory remain available
under `inventory` without changing the public compatibility fields.

The report is releasable only when `supported` is `true`, the summary has zero
`missing`, `insufficient`, and `invalid` entries, its public schema validation
succeeds, and regenerating and formatting it with the commands above from the
recorded catalog commit produces the same bytes. A failed report remains useful
during development because it names every gap without turning it into a silent
success.
